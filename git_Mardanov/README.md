Сканер вирусов VirusTotal.

1\. Общие положения:

1.1 Полное наименование системы: «Сканер файлов VirusTotal».

1.2 Шифр темы: Учебный проект МДК 03.01, СПб ГУАП, 2025 г.

1.3 Предприятие-разработчик системы: Марданов Андрей, Казакевич Даниил, Патрушев Тимофей.

1.4 Предприятие-заказчик системы: Кафедра информационных систем и программирования, Федеральное государственное автономное образовательное учреждение высшего образования «Санкт-Петербургский государственный университет аэрокосмического приборостроения».

1.5 Система создается на основании технического задания (ТЗ): Требуется разработать настольное приложение для Windows, реализующее:

Загрузку файлов пользователя на сервис VirusTotal.

Автоматическое ожидание завершения анализа.

Получение и визуализацию результатов сканирования в удобной для пользователя форме.

Отображение статистики по детектированию вредоносного, подозрительного и безопасного ПО.

1.6 Плановый срок начала работ: 10.09.2025

1.7 Плановый срок окончания работ: 18.09.2025

1.8 Архитектура системы: Клиентское приложение (WPF, C#, .NET Framework 4.7.2+), взаимодействующее с внешним облачным API (VirusTotal).

1.9 Порядок оформления и предъявления заказчику результатов работы:\*\* Программа считается принятой после успешного прохождения тестов, подтверждающих выполнение всех функций, перечисленных в разделе 4.1.

2\. Назначение и цели создания системы:

2.1 Вид автоматизируемой деятельности:

Автоматизация процесса проверки файлов на наличие вредоносного ПО с использованием агрегированного анализа множества антивирусных движков через единый интерфейс.



2.2 Перечень автоматизируемых процессов:

2.2.1 Для пользователя (студента, преподавателя, ИТ-специалиста):

Выбор файла для проверки через графический интерфейс.

Отправка выбранного файла на сервер VirusTotal.

Автоматическое отслеживание статуса анализа без необходимости ручного обновления.

Получение итогового отчета с детализацией по результатам сканирования.

Визуальная индикация уровня угрозы (опасно, подозрительно, чисто).

2.3 Наименование и значение показателей, которые будут достигнуты в результате внедрения системы:

2.3.1 Удобство использования:

Время на запуск проверки одного файла — не более 10 секунд.

Интуитивно понятный интерфейс, не требующий специальных знаний.

2.3.2 Оперативность получения информации:

Результаты анализа отображаются сразу после их готовности на стороне VirusTotal.

Максимальное время ожидания результата — 180 секунд (3 минуты).

2.3.3 Надежность и точность:

Система корректно обрабатывает ошибки (недоступность сети, неверный API-ключ, отмена пользователем).

Все результаты анализа отображаются в неизменном виде, как они предоставлены API VirusTotal, гарантируя точность данных.

2.3.4 Доступность:

Приложение работает автономно на ПК пользователя, зависимость только от доступа в интернет для связи с API.

3\. Характеристики объекта автоматизации:

3.1 Краткие сведения о пользователях:

Основными пользователями являются студенты, преподаватели и ИТ-специалисты СПб ГУАП, которым требуется быстрый и надежный способ проверить подозрительные файлы, полученные из интернета, по электронной почте или с внешних носителей.

3.2 Процесс проверки файлов характеризуется:

Необходимостью использовать множество антивирусных решений для повышения точности детектирования.

Сложностью и временными затратами при ручной загрузке файлов на веб-сайт VirusTotal.

Потребностью в быстрой интерпретации результатов большого количества сканеров.

3.3 Особенности объекта автоматизации:

Необходимость интеграции с внешним, асинхронным API.

Требование к приложению быть простым, быстрым и не требующим установки сложного серверного ПО.

Акцент на безопасности и прозрачности результатов для конечного пользователя.

3.4 Внедрение системы направлено на:

Повышение уровня кибербезопасности пользователей университета.

Сокращение времени и усилий, затрачиваемых на ручную проверку файлов.

Обеспечение единого, стандартизированного подхода к анализу подозрительных файлов.

4\. Требования к ИС:

4.1 Требования к системе в целом:

4.1.1 Система должна обеспечивать сквозной процесс: от выбора файла до отображения финального отчета.

4.1.2 Интерфейс должен быть интуитивно понятным и не перегруженным лишней информацией.

4.1.3 Приложение должно корректно обрабатывать сетевые ошибки и ошибки API, информируя пользователя понятными сообщениями.

4.1.4 Доступность системы зависит только от наличия интернет-соединения на клиентской машине.



4.2 Требования к программному обеспечению ИС:

4.2.1 Клиентская часть:

Технология: WPF (Windows Presentation Foundation).

Язык программирования: C#.

Целевая платформа: .NET Framework 4.7.2 и выше.

Операционная система: Windows 10/11.

4.2.2 Функциональность:

Загрузка файла через диалоговое окно.

Отправка файла на `https://www.virustotal.com/api/v3/files`.

Периодический опрос статуса анализа по ID.

Получение и парсинг JSON-ответа с результатами.

Форматирование и отображение статистики (вредоносные, подозрительные, безвредные и т.д. сканеры).

Визуальное выделение итогового вердикта (опасно/подозрительно/чисто).

4.3 Требования к техническому обеспечению АС:

4.3.1 Клиенты:

Операционная система: Windows 10 или Windows 11.

Процессор: Pentium 4 и выше (рекомендуется Intel i3 и выше).

Оперативная память: 2 ГБ и выше (рекомендуется 4 ГБ и выше).

Свободное место на диске: 50 МБ.

Обязательное условие: постоянное подключение к сети Интернет.

5\. Стадии и этапы разработки:

5.1 Разработка должна быть проведена в три стадии:

5.1.1 Разработка технического задания (ТЗ) — формирование требований, определение функциональности и критериев приемки.

5.1.2 Рабочее проектирование и разработка — создание архитектуры приложения, проектирование интерфейса, написание кода, интеграция с API VirusTotal.

5.1.3 Тестирование и сдача — комплексное тестирование всех функций, устранение ошибок, передача исполняемого файла и документации заказчику.

6\. Этапы разработки:

6.1 На стадии рабочего проектирования и разработки должны быть выполнены следующие работы:

Создание макета пользовательского интерфейса (UI/UX).

Реализация логики загрузки и отправки файлов.

Реализация механизма ожидания и получения результатов анализа.

Разработка модуля форматирования и отображения результатов.

Реализация обработки исключений и ошибок.

Компиляция и сборка исполняемого файла (.exe).

6.2 На этапе тестирования и сдачи:

Проведение модульного и интеграционного тестирования.

Проверка корректности работы с различными типами файлов и в условиях сетевых сбоев.

Передача финальной версии программы (.exe) и сопроводительной документации (руководство пользователя) заказчику.

Демонстрация работоспособности приложения и подписание акта приемки.
